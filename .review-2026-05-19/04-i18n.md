# 04-i18n

## Summary
i18n.js carries two parallel PT/EN dictionaries plus `t()` / `tSkill()` / `tJournal()` / `tJournalCat()` helpers, and the wiring on the dashboard, skills, quests, journal and lookup pages is healthy. The rot is concentrated outside the dictionary: ~50 user-facing strings in goals.js, money.js, live.js, next-steps.js and combat.js are hardcoded behind `lang === "pt"` ternaries (bypassing the dictionary), one `t()` call (`easterTitle`) has no key in either language, the dictionary has duplicate (`updated:`) and orphan (`meetup*`, `sortId/Gap/Alpha/Xp`, `decMonth`, `gainsTitle`, `noReqs`, `skills`, `quests`) keys, and a few static HTML strings (`#af-other`, skill-sort buttons, `<title>`, default subtitle) only get localized after JS boots, leaving a brief PT-only flash for first-paint EN users.

## Severity legend
P0 = breaks core feature, P1 = significant bug/UX flaw, P2 = polish

## Findings

### [P1] `t("easterTitle")` references a key that does not exist
- **Where:** `script.js:1217` (`s("easter-title", t("easterTitle"));`); also `script.js:1218` builds `"Blooming Burrow · 30 Mar - 20 " + t("aprMonth")`.
- **What:** No `easterTitle` key in either `LANG.pt` or `LANG.en` (grep `easterTitle` in `i18n.js` returns 0 results). `t()` falls back to the literal key string, so if `#easter-title` ever exists in markup it will render the raw string `"easterTitle"`.
- **Why:** Key was either renamed, removed, or never added. The `#easter-title` / `#easter-sub` elements aren't in `index.html` today, so `s()` silently no-ops — the bug is latent and will surface the next time the Easter section is re-added.
- **Fix:** Either add `easterTitle` to both PT/EN dictionaries or delete the two `s()` calls in `updateUIText()`. The neighbouring `aprMonth` / `decMonth` pair has the same smell — `decMonth` is defined but never read.

### [P1] Most goals.js / money.js / live.js strings bypass i18n
- **Where:**
  - `goals.js:299, 310, 323, 359, 387, 392, 397, 412, 450, 473, 540, 542, 545, 547, 550, 556, 563, 573, 579, 587, 595, 604` (~24 inline strings).
  - `money.js:737, 738, 777, 778, 780, 801, 802, 914, 916, 917, 922, 923, 924, 928, 934, 939, 947, 949, 950, 994` (~20).
  - `live.js:266–429` — every label/string is built with `lang === "pt" ? ... : ...` (40+ literals: "Sessão", "Nível", "ETA", "Atualizar", "Combate", "Top 3 da sessão", "polling pausado", etc.).
  - `next-steps.js:227, 247` (PT/EN inlined via `.pt`/`.en` fields on data rows; the helper `nsT()` shadow-dictionary at `next-steps.js:151–172` defines 16 keys that should live in `i18n.js`).
  - `major-goals.js:11–20` — 9 strings live in a local `mgT()` map shadowing `t()`.
  - `combat.js:961, 966` — `"Aprimorada"` / `"Enhanced"` / `"Básica"` / `"Basic"` inline; `combat.js:887, 977` render `"Lv."` (untranslated, but at least neutral).
- **What:** ~50 user-facing strings duplicate the PT/EN pattern outside `LANG.*`. Several reinvent keys that already exist (`t("daily")` = "DIÁRIO"/"DAILY" vs money.js inline `"Diário"`/`"Daily"` at line 737; `t("soon")` vs `"Quase"`/`"Soon"` at line 738; `t("skills")`/`t("quests")` vs goals.js inline `"Hab"`/`"Skills"` at line 387 and `"Missões"`/`"Quests"` at line 392; `t("complete")`/`t("started")`/`t("remaining")` vs the lookup/script versions). When PT copy changes, the maintainer has to find each clone.
- **Why:** Pages were added module-by-module; each author rolled their own shortcut rather than extending the central dictionary. `mgT()` (major-goals.js) and `nsT()` (next-steps.js) at least scope the strings, but goals.js / money.js / live.js sprinkle raw ternaries through templates.
- **Fix:** Promote the keys into `LANG.pt` / `LANG.en`. As a stop-gap, prefer `nsT()`-style local maps over raw ternaries so the strings can be audited. Suggested new keys (names indicative): `nsTitle`, `nsSub`, `nsNextUnlock`, `nsNextQuest`, `nsDaily`, `mnDailyRoutine`, `mnPowerLoops`, `mnLivePrices`, `mnCachedPrices`, `mnAvailable`, `mnAlmost`, `mnSort`, `mnNoMethods`, `glWhatsMissing`, `glPath`, `glSkillsToTrain`, `glQuestsToComplete`, `glPendingItems`, `glAlreadyDone`, `glGoalComplete`, `glCapstone`, `glNextActions`, `liveSession`, `liveTotalXp`, `liveTop3`, `liveBaseline`, `liveResetBaseline`, `liveTips`, `liveCacheHint`.

### [P1] Duplicate `updated:` key in both PT and EN
- **Where:** `i18n.js:42` and `i18n.js:190` (PT), `i18n.js:335` and `i18n.js:477` (EN).
- **What:** `updated` is declared twice in each language object literal. JavaScript silently keeps the last definition; both values happen to be identical (`"Atualizado"` / `"Updated"`), so no runtime symptom today, but any future edit that touches one of the two will be lost without warning.
- **Why:** Footer block (line 187+) was added later and re-declared the key from the nav block (line 42).
- **Fix:** Delete the second occurrence in each language, leaving a single source of truth. Linters / `eslint`'s `no-dupe-keys` would have caught this.

### [P2] Orphan keys: `meetup*`, sort labels, `gainsTitle`, `decMonth`, `noReqs`, top-level `skills`/`quests`
- **Where:** `i18n.js:114–115, 126–135, 81–84, 142, 44, 32, 184–185` (and EN parallels).
- **What:** None of the following keys are reached by any `t()` / `tSkill()` / `tJournalCat()` call (verified via grep across the repo):
  - `meetupCompleted`, `meetupNoPlanned`, `meetupXpPerSkill`, `meetupTotalXp`, `meetupTotalLevels`, `meetupResult`, `meetupSessionSummary`, `meetupXpGained`, `meetupHighlights`, `meetupEmpty`, `meetupChartsEmpty`, `meetupLevelsGained` (entire Meetup block).
  - `sortId`, `sortGap`, `sortAlpha`, `sortXp` — the skill-sort buttons (`index.html:127–130`) hardcode the English labels and `updateUIText()` never overwrites them. Visually fine because the labels are language-neutral abbreviations.
  - `gainsTitle`, `decMonth`, `noReqs`, `visits`, `progress`, `clueScrolls` vs `clueScrollsLabel` (overlap).
  - Top-level `skills:` (`i18n.js:184`) and `quests:` (`i18n.js:185`) — shadowed by `journalCats.skills` / `journalCats.quests` which is what `tJournalCat()` actually reads.
- **Why:** Features were removed (Meetup section) without sweeping the dictionary; HTML didn't adopt sortable button localization.
- **Fix:** Either wire the keys (e.g. `s("sort-default", t("sortId"))` etc., feed sort buttons through `updateUIText()`) or delete them. Recommend a quick audit and pruning pass — the dictionary is the user-facing surface and orphans make missing-key bugs harder to spot.

### [P2] First-paint flash: HTML defaults are PT-only, with one mismatch
- **Where:**
  - `index.html:2` — `<html lang="pt-BR">` (correct only for PT users).
  - `index.html:6` — `<title>RS3 Placar — Fiorovizk & Decxus</title>` PT default.
  - `index.html:32` — `<p class="subtitle">Acompanhamento em tempo real</p>` — note this is **different copy** from the PT `subtitle` key (`"Jornada cooperativa em tempo real"`, `i18n.js:10`). EN users see the HTML default ("Acompanhamento em tempo real") for ~one frame, then a different PT string ("Jornada cooperativa em tempo real") for one more frame as `updateUIText()` runs synchronously after `setLang`, then finally the EN string. PT users see a single re-write within the same frame.
  - `index.html:99–102` — activity filter buttons hardcode `"All"`, `"Level-ups"`, `"Quests"`, `"Bosses"`. No `#af-other` button despite `script.js:1209` updating it (button missing, key wasted).
  - `index.html:120–124, 127–130, 157–161, 172–176` — pill filter buttons default to mixed PT/EN.
- **What:** Cold load on a saved-EN user shows ~200ms of PT before JS boots. Worse, the HTML default subtitle copy diverges from the i18n entry, so PT users see a brief swap between two PT phrasings.
- **Why:** Static HTML wasn't matched to the `pt` dictionary as the dictionary evolved; no SSR / inline-bootstrap step writes the lang-correct defaults.
- **Fix:**
  - Sync `index.html:32` to the PT `subtitle` value (or update the dictionary to match the HTML).
  - Add a tiny inline script in `<head>` that reads `localStorage["rs3lb-lang"]` and toggles `<html lang>` before render to cut the flash for screen readers; the visible text is already overwritten by `updateUIText()` shortly after.
  - Drop the unused `af-other` translation OR add the missing button to `#activity-filters`.

### [P2] `t("title").replace("RS3 ", "")` is brittle
- **Where:** `script.js:1150`.
- **What:** Strips the literal `"RS3 "` prefix from the localized title to recover the "Placar" / "Leaderboard" suffix for the `<span class="accent">`. Works today because both PT (`"RS3 Placar"`) and EN (`"RS3 Leaderboard"`) start with `"RS3 "`, but any future localized title that drops the prefix (e.g. an i18n update to `"Placar RS3"`) silently leaves the accent span identical to the full title.
- **Fix:** Split into two keys: `brand` ("RS3", language-neutral) and `titleSuffix` ("Placar" / "Leaderboard"), then render `${t("brand")} <span class="accent">${t("titleSuffix")}</span>`.

### [P2] Combat string-split locale shortcut
- **Where:** `combat.js:893` — `DPS (${L.single.split(" ")[0]})`.
- **What:** Re-uses the localized "Single Target" / "Alvo Único" string and pulls the first word as a short DPS label. PT split → "Alvo"; EN split → "Single". Functional but fragile — any single-word PT/EN translation (e.g. `"Alvo-Único"`) would break the label.
- **Fix:** Add a dedicated key `cbDpsShort` (PT: `"Alvo"`, EN: `"Single"`) and use it instead of `.split(" ")[0]`. Same applies to the hardcoded `"AoE"` literal at `combat.js:897` (matches `cbAoe` already in the dictionary — use `t("cbAoe")` for consistency).

### [P2] Combat ability-type labels are double-coded
- **Where:** `combat.js:959–968` — `"Aprimorada"`/`"Enhanced"`/`"Básica"`/`"Basic"` inline; `"Ultimate"` hardcoded both ways.
- **What:** Ability tooltips bypass the dictionary. Three keys would centralize these and let translators tweak them without code edits.
- **Fix:** Add `cbAbilityBasic`, `cbAbilityEnhanced`, `cbAbilityUltimate` to the dictionary; replace the ternary at `combat.js:959–968` with `t(...)`.

### [P2] `Members` / `F2P` left untranslated
- **Where:** `money.js:940–941` — `⭐ Members`, `F2P` buttons.
- **What:** These are universally recognized in the RS community but inconsistent with the rest of the localized filter row (`Disponíveis`/`Quase`/`Todas`). Low impact.
- **Fix:** Add `mnMembers` / `mnF2P` keys for parity, or document the intentional non-translation in a comment.

### [P2] `jfilter-xp` button has no localization hook
- **Where:** `index.html:176` (button `#jfilter-xp`), `script.js:1192–1195` (no `s("jfilter-xp", tJournalCat("xp"))`).
- **What:** All other journal filter buttons are re-labelled by `updateUIText()`; the XP one isn't. The label "XP" is language-neutral so users won't notice, but the inconsistency makes the code harder to reason about.
- **Fix:** Add `s("jfilter-xp", tJournalCat("xp"));` after line 1195.

### [P2] `qww` vs `qww2` journal-goal key collision risk
- **Where:** `i18n.js:271, 290` (PT) — `qww` = Wolf Whistle, `qww2` = The World Wakes.
- **What:** Two distinct quest journals share a 3-character prefix. Any future contributor adding a "Wolf Whistle II" / "The World Wakes II" goal would naturally pick `qww2`, silently shadowing the existing entry. The current key naming convention everywhere else uses unique abbreviations (`qhg`, `qrm`, `qpe`, `qwgs`).
- **Fix:** Rename `qww2` to `qtww` (The World Wakes initialism) for clarity. Touch sites: `i18n.js` (both langs) plus any goal-id reference in goals.js / journal data.

### [P2] `clueScrolls` vs `clueScrollsLabel` — pick one
- **Where:** `i18n.js:154` (`clueScrolls`) used at `script.js:710`; `i18n.js:93` (`clueScrollsLabel`) used at `script.js:734`.
- **What:** Two keys for the same concept. Both happen to translate to the same PT string ("Pergaminhos") and the same EN string ("Clue Scrolls"), so they're already inconsistent-as-a-pair. Anyone updating PT copy must remember to touch both.
- **Fix:** Collapse into a single `clueScrolls` key, delete `clueScrollsLabel`, update `script.js:734`.

## Quick wins (ordered by ROI)
1. Remove duplicate `updated:` keys in both PT and EN (3-line delete) — eliminates a silent-loss footgun.
2. Add `easterTitle` key or remove the two `s("easter-title", ...)` / `s("easter-sub", ...)` lines (`script.js:1217–1218`) — kills the latent literal-string-leak.
3. Sync `index.html:32` subtitle to match the `subtitle` key value (`"Jornada cooperativa em tempo real"`) — kills the double-flash on PT cold load.
4. Replace the most-visible inline ternaries in `live.js` (Session / Total XP / ETA / Refresh / Reset baseline / Top 3 — `live.js:401–429`) with dictionary keys — Live tab is one of the busiest screens and currently has the most untranslated UX.
5. Audit & prune the orphan `meetup*` block plus `gainsTitle`, `decMonth`, `visits`, `progress`, top-level `skills`/`quests` keys.
6. Add `t()` plumbing for `#skill-sort` buttons (`index.html:127–130` ↔ existing `sortId/Gap/Alpha/Xp` keys) — finishes a half-built feature.
7. Promote `nsT()` (next-steps.js) and `mgT()` (major-goals.js) maps into `LANG.*` so all keys live in one file.
8. Split title into `brand` + `titleSuffix` to remove the `.replace("RS3 ", "")` fragility.

## Notes / open questions
- The activity-text PT localization in `script.js:1100–1140` is regex-based on the EN strings returned by RuneMetrics. Works for two languages, but adding a third (or RuneMetrics changing wording) would require a parallel rule set. Not a current bug — flagged for awareness.
- `setLang()` in `i18n.js:597–607` correctly keeps `<html lang>` and the toggle's `aria-pressed` in sync, but only when invoked from the toggle. On cold load, the same wiring happens inside `updateUIText()` (`script.js:1152–1153`). Two code paths doing the same DOM update is fine functionally, but a future refactor should converge on `setLang()` calling `updateUIText()` (or vice versa) to avoid divergence.
- `tJournal(goalId)` returns `{ title: goalId, desc: "" }` as the final fallback (`i18n.js:589`). Reasonable. Worth confirming every `GOALS[].id` in `goals.js` has a matching `journal.<id>` entry — a missing entry shows a raw goal-id slug in the UI.
- Should the bilingual data-row pattern in `next-steps.js:12–72` (`u_pt`/`u_en`/`hint_pt`/`hint_en`) and `goals.js:14–17` (`label_pt`/`label_en`/`sub_pt`/`sub_en`) migrate into the dictionary, or stay alongside their data? Inline arguably keeps the per-row context, but it makes a 3rd-language addition far more invasive. Owner call.
- No automated check exists for "every key present in PT is present in EN, and vice versa". A 20-line dev-time assertion in i18n.js (or a tiny test script) would catch the next missing-key bug at load time instead of in production.
